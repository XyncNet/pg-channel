# pg-channel

> [English version](./README.md)

Генератор готовых к запуску PL/pgSQL-скриптов, которые поднимают каналы PostgreSQL `LISTEN`/`NOTIFY` на события `INSERT` / `UPDATE` / `DELETE` для указанной таблицы.

## Зачем нужен проект

`LISTEN`/`NOTIFY` в PostgreSQL — самый дешёвый способ доставлять изменения строк в приложения (воркеры, websocket-сервисы, инвалидацию кеша и т. п.) без опроса БД. Но писать триггерные функции вручную утомительно и легко ошибиться:

- на каждую таблицу и каждое событие нужна отдельная триггерная функция;
- формат полезной нагрузки должен совпадать во всех сервисах-потребителях канала;
- при `UPDATE` обычно нужна не вся строка целиком, а **diff** — какие поля изменились и с какого значения на какое;
- часто требуются условные каналы — например, нотификация только при изменении конкретных полей, с логикой `AND` / `OR` между ними.

`pg-channel` решает всё это одной функцией `plsql()`: по имени таблицы и (опционально) словарю условий апдейта она возвращает готовый PL/pgSQL-скрипт. Достаточно выполнить его в БД один раз — и каналы заработают.

### Соглашения о полезной нагрузке каналов

- `tables_new` — на `INSERT`, нагрузка = полная новая строка как JSON (`row_to_json(NEW)`).
- `tables_del` — на `DELETE`, нагрузка = `OLD.id` (строкой).
- `tables_upd` (или `tables_upd_<event>`) — на `UPDATE`, нагрузка = JSON-массив `[NEW.id, diff]`, где `diff` — объект вида `{field_name: [old_value, new_value], ...}` **только из изменённых полей**. Если ничего не изменилось, `diff` будет `null`.

> Канал `del` намеренно отправляет только id — у удалённой строки больше нет осмысленного состояния для потребителя, а числовой id экономит размер сообщения.

```bash
pip install pg-channel
```

## Использование

```python
from pg_channel import plsql, Act
```

`plsql(table, ops=7, updates=None)` возвращает строку с полным PL/pgSQL-скриптом. Выполнить его в БД можно любым клиентом (`psycopg`, `asyncpg`, `psql -f`, инструмент миграций и т. д.).

### Аргументы

| имя       | тип                            | по умолч. | описание                                                                                                                                                |
|-----------|--------------------------------|-----------|---------------------------------------------------------------------------------------------------------------------------------------------------------|
| `table`   | `str`                          | —         | Имя таблицы. Используется как префикс канала (`{table}s_*`) и оборачивается в двойные кавычки внутри `CREATE TRIGGER`.                                  |
| `ops`     | `int` (битовая маска `Act`)    | `7`       | Какие триггеры генерировать. `Act.DEL=1`, `Act.NEW=2`, `Act.UPD=4`. Комбинируются побитовым ИЛИ. `7` = все три.                                          |
| `updates` | `dict[str, tuple \| list]`     | `None`    | Опциональная маршрутизация для апдейт-триггера. Ключи — имена событий, значения — наборы полей. См. [Условные каналы апдейта](#условные-каналы-апдейта). |

### Примеры

#### Все три триггера, один общий канал апдейта

```python
plsql("my_ad")
```

Запускает:
- `my_ads_new` — при вставке,
- `my_ads_del` — при удалении,
- `my_ads_upd` — при любом обновлении.

#### Только апдейт-триггер

```python
plsql("my_ad", Act.UPD)   # то же самое, что plsql("my_ad", 4)
```

Сгенерированная функция апдейта выглядит так:

```sql
CREATE OR REPLACE FUNCTION my_ad_upd() returns trigger as $my_ad_upd_trg$
    DECLARE
        diff jsonb;
    BEGIN
        SELECT jsonb_object_agg(key, jsonb_build_array(to_jsonb(OLD) -> key, value))
        INTO diff
        FROM jsonb_each(to_jsonb(NEW))
        WHERE value IS DISTINCT FROM (to_jsonb(OLD) -> key);

        PERFORM pg_notify('my_ads_upd', jsonb_build_array(NEW.id, diff)::varchar);
        RETURN NULL;
    END
    $my_ad_upd_trg$ LANGUAGE plpgsql;
CREATE OR REPLACE TRIGGER my_ad_upd AFTER UPDATE ON "my_ad" FOR EACH ROW EXECUTE FUNCTION my_ad_upd();
```

Подписчик канала `my_ads_upd` будет получать сообщения вида:

```json
[42, {"status": ["draft", "published"], "price": [100, 120]}]
```

### Условные каналы апдейта

Аргумент `updates` разбивает единый канал `tables_upd` на несколько именованных подканалов, каждый из которых срабатывает только при выполнении своего условия по полям.

- **Ключ** — имя события; имя канала становится `{table}s_upd_{event}`.
- **Значение** — поля, по изменению которых событие срабатывает:
  - `list` → **OR** (хотя бы одно из перечисленных полей изменилось),
  - `tuple` → **AND** (все перечисленные поля изменились).
- **Ключ с префиксом `_`** → ветка генерируется как `ELSIF`, а не как самостоятельный `IF`, и срабатывает только если не сработала ни одна предыдущая ветка (удобно для взаимоисключающих событий).

#### Одно конкретное поле

```python
plsql("order", 7, {"change": ["status"]})
```

Запускает `orders_upd_change` только при изменении `order.status`.

#### Смешанный AND / OR

```python
plsql("table3", 5, {
    "event1": ("field1", "field2"),   # tuple → AND
    "event2": ["field2", "field3"],   # list  → OR
})
```

Сгенерируются два независимых `IF`-блока — оба могут сработать на одном обновлении.

#### Взаимоисключающие ветки (`ELSIF`)

```python
plsql("order", 4, {
    "published": ["status"],
    "_priced":   ["price"],   # префикс "_" → ELSIF
})
```

Если изменился `status`, сработает `orders_upd_published`, а ветка по `price` будет пропущена, даже если `price` тоже поменялся.

### Подписка из Python

```python
import asyncpg, asyncio, json

async def main():
    conn = await asyncpg.connect(dsn=...)
    await conn.add_listener("my_ads_upd", lambda c, pid, ch, payload: print(ch, json.loads(payload)))
    await asyncio.Future()   # держим соединение открытым

asyncio.run(main())
```

## Нюансы и ограничения

- **Подразумевается колонка `NEW.id`.** В триггере апдейта вызывается `jsonb_build_array(NEW.id, diff)`, в триггере удаления — `OLD.id`. Для таблиц без `id` придётся править сгенерированный SQL вручную.
- **Множественное число у канала — наивное.** `plsql("my_ad")` даёт `my_ads_*` — библиотека просто добавляет `s`. Подбирайте имя таблицы так, чтобы такое «множественное число» выглядело разумно, или переименовывайте каналы после генерации.
- **Имя таблицы экранируется, имя канала — нет.** `CREATE TRIGGER` использует `"{table}"` (так что зарезервированные слова вроде `order` безопасны), а имена каналов — «как есть»; ограничьтесь идентификаторами в нижнем регистре.
- **`row_to_json` против `to_jsonb`.** Канал `NEW` использует `row_to_json` (insert-нагрузка), а diff в `UPD` — `to_jsonb`. Оба сериализуют типы PostgreSQL в JSON, но проверьте, как ваши колонки (особенно `bytea`, кастомные enum, range-типы) выглядят в JSON, прежде чем закладываться на формат.
- **Лимит на размер payload.** По умолчанию каждый `NOTIFY` ограничен 8000 байт. Широкие строки или большие текстовые поля могут не влезть — используйте узкие каналы (через `updates`) или передавайте только id, а данные перезапрашивайте на стороне потребителя.
- **`CREATE OR REPLACE`.** И функции, и триггеры создаются через `CREATE OR REPLACE`, поэтому повторный запуск скрипта идемпотентен.
- **Тайминг триггеров.** Все триггеры — `AFTER` и `RETURN NULL`, они не влияют на саму операцию.
